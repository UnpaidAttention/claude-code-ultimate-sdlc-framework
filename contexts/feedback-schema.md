# Feedback Subsystem — Schema & Conventions

Authoritative reference for the feedback subsystem introduced in the Claude Code Ultimate SDLC Framework. Governance rules live in `rules/feedback-rules.md` (P0). This document is the schema, naming, and file-layout reference that skills and workflows consult.

---

## 1. Purpose

Capture **how the team wants work done** (corrections + reasoning), separately from **what the system does** (specs). Enable the AI to apply learned preferences in future sessions, cycle-over-cycle, without re-learning the same lessons.

Requirements = specs. Feedback = preferences + corrections + rule gaps. They do not overlap.

---

## 2. File Layout (Per-Project)

```
.ultimate-sdlc/feedback/
├── INDEX.md                          # Single index; entries grouped by status
├── FB-001-<slug>.md                  # Individual entries; zero-padded monotonic IDs
├── FB-002-<slug>.md
├── ...
└── REJECTED.md                       # Append-only log of entries rejected at write time

.ultimate-sdlc/framework-revisions-proposed/
├── FR-001-<slug>.md                  # Framework revision proposals — surfaced to user
├── FR-002-<slug>.md                  # NEVER auto-applied to the framework repo
└── ...
```

- `FB-NNN` = Feedback entry. IDs are monotonic across all four types.
- `FR-NNN` = Framework revision proposal. Separate ID space from FB.
- Slugs are lowercase-kebab-case, ≤6 words.

---

## 3. Feedback Types

| Type | Purpose | Written by | Read by |
|------|---------|------------|---------|
| `user-correction` | User corrected an agent output + gave reason | `/sdlc-feedback-log` | Session start, pre-AIOU |
| `user-preference` | User stated a preference unprompted by an error | `/sdlc-feedback-log` | Session start, pre-AIOU |
| `gate-learning` | A gate failure revealed a rule/process gap | `/sdlc-feedback-log --type gate-learning` OR gate workflow | Pre-gate check |
| `pattern` | Cycle-end synthesis of ≥2 recurring entries | `/sdlc-feedback-promote` | New cycle start |

---

## 4. Feedback Entry Frontmatter Schema

| Field | Required | Type | Constraint |
|-------|----------|------|-----------|
| `id` | yes | string | `FB-NNN` where NNN is zero-padded 3 digits, monotonic across all types |
| `name` | yes | string | Lowercase-kebab-case slug, ≤6 words |
| `description` | yes | string | Single line, ≤120 chars. Used in INDEX.md |
| `type` | yes | enum | `user-correction` \| `user-preference` \| `gate-learning` \| `pattern` |
| `council` | yes | enum | `planning` \| `development` \| `audit` \| `validation` \| `any` |
| `phase` | yes | string | Phase/wave/track identifier, or `any` |
| `cycle` | yes | string | Cycle ID this was recorded in |
| `created` | yes | string | ISO 8601 timestamp |
| `updated` | yes | string | ISO 8601 timestamp |
| `status` | yes | enum | `active` \| `resolved` \| `superseded` \| `promoted` |
| `tags` | no | array | Domain tags for matching (e.g. `[validation, email]`) |
| `supersedes` | no | array | IDs of entries this one replaces |
| `promoted_to` | no | string \| null | If `status: promoted`, the pattern FB-NNN that consumed this |
| `references` | no | array | Related spec/artifact IDs (FEAT-NNN, AIOU-NNN, gate-IDs) |
| `user_reasoning_source` | yes | enum | `verbatim` \| `paraphrase` \| `inferred` |

**Validation**: `/sdlc-feedback-log` validates all required fields and that `type` values are in-enum. Invalid entries are rejected.

## 5. Required Body Sections

Every feedback entry body MUST contain these sections, in order:

1. **What was wrong** — factual description of the agent's action or draft that triggered the correction. No blame language.
2. **Fix applied** — what the corrected version is (code, spec edit, or behavior).
3. **Why (user's reasoning)** — the user's explanation. **Empty or missing = invalid entry (FBP-001).**
4. **How to apply (generalized guidance)** — actionable "When X, do Y because Z" for future sessions.
5. **Scope of applicability** — councils, phases, trigger conditions.
6. **Anti-weaponization check** — explicit checkboxes for PRH-001..PRH-009. All must pass.

---

## 6. Index Structure (`INDEX.md`)

Groups entries by status:
- `Active` — currently applied
- `Promoted` — rolled into a pattern
- `Superseded` — replaced by newer entry
- `Resolved` — underlying issue no longer applies

Each line:
```
- [FB-NNN](FB-NNN-slug.md) `type` `council/phase` — description
```

Plus a stats table and a `Next ID` pointer.

---

## 7. Framework Revision Proposals

Generated by `/sdlc-framework-retro` at cycle close. Saved to `.ultimate-sdlc/framework-revisions-proposed/` (NOT the framework repo). Structure per `templates/framework-revision.md`:

- Frontmatter: `id`, `target_file`, `target_section`, `impact`, `reversibility`, `evidence`, `status`
- Body: Current state excerpt, Proposed change (diff), Rationale, Evidence trail, Risk assessment
- **Absolute rule**: The framework repo is never auto-modified. Proposals are surfaced to the user; the user applies or rejects.

---

## 8. ID Allocation

- **FB IDs** are per-project, stored in `.ultimate-sdlc/feedback/INDEX.md § ID allocation`.
- **FR IDs** are per-project, stored in `.ultimate-sdlc/framework-revisions-proposed/INDEX.md § ID allocation` (created on first use).
- IDs never reuse — even if an entry is resolved/rejected/superseded, its ID is retired.

---

## 9. Lifecycle States

```
write → active
active → superseded (when newer entry replaces)
active → resolved (when underlying issue gone)
active → promoted (when rolled into pattern at cycle end)
```

Transitions are one-way. No entry goes from `superseded` back to `active` — create a new entry instead.

---

## 10. Read Filters (How Skills Query Feedback)

Skills that consume feedback filter by:

| Filter | Applies to |
|--------|-----------|
| `status == active` | Always (never read non-active for application) |
| `council in {current_council, any}` | Session start load |
| `phase in {current_phase, any}` | Pre-AIOU / pre-gate load |
| `references contains {AIOU_id}` | Context Load step of AIOU execution |
| `tags intersects {AIOU_tags}` | Context Load step of AIOU execution |
| `type == gate-learning AND references contains {gate_id}` | Pre-gate verification |

---

## 11. Rejection Log (`REJECTED.md`)

When `/sdlc-feedback-log` rejects an entry (anti-weaponization check fails, missing reasoning, etc.), append to `.ultimate-sdlc/feedback/REJECTED.md`:

```markdown
## <ISO 8601 timestamp> — rejection reason

**Attempted type:** <type>
**Council/phase:** <council/phase>
**Why rejected:** <FBP-NNN or FR-NNN violated, specific reason>
**Agent's suggested alternative:** <e.g. "this belongs in FEAT-XXX spec update, not feedback">
```

This log itself is non-authoritative — it's a trail to help the user understand why certain corrections didn't become feedback entries.

## 12. Integration Points Summary

| Integration | File / Workflow | Action |
|-------------|-----------------|--------|
| Session start | All council rules, Session Protocol | Read INDEX.md → load `active` entries matching current council |
| Pre-AIOU (Planning 2.5/3/3.5, Dev waves) | council-planning.md, council-development.md | Filter by tags → apply to implementation plan |
| Pre-gate | Each gate workflow | Load `gate-learning` for this gate → verify prevention applied |
| Cycle close | `/sdlc-close-cycle`, `/sdlc-framework-retro` | Synthesize patterns + draft revision proposals |
| New cycle | `/sdlc-new-cycle` | Load carried-forward `pattern` entries |
| Validation S1 | council-validation.md | Invoke `/sdlc-feedback-promote` |

---

## 13. Non-Goals

This subsystem does NOT:
- Propagate feedback across projects (by design — FR-4)
- Auto-modify the framework repo (by design — FR-3)
- Replace specs (FR-1 — specs describe WHAT, feedback describes HOW)
- Override INTEGRITY-RULES (FR-2 — anti-weaponization prohibits bypass)
